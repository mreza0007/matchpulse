# معماری پیشنهادی MatchPulse v2

این سند یک نقشه راه اولیه برای تبدیل MatchPulse از یک مینی‌اپ مخصوص جام جهانی ۲۰۲۶ به یک پلتفرم حرفه‌ای فوتبال زنده، پیش‌بینی و لیدربورد است. هدف این مرحله فقط مستندسازی مسیر آینده است و نباید باعث refactor، تغییر رفتار فعلی یا شکستن تجربه فعلی جام جهانی شود.

## ۱. جهت‌گیری محصول

MatchPulse v2 باید از حالت تک‌رقابتی خارج شود و چند رقابت مهم فوتبال را پشتیبانی کند:

- Premier League
- UEFA Champions League
- LaLiga
- Serie A
- Bundesliga
- Euro
- CAFA
- World Cup

محصول نهایی باید ترکیبی از این تجربه‌ها باشد:

- دنبال‌کردن بازی‌های زنده و نتایج
- مشاهده برنامه مسابقات، جدول‌ها و رویدادهای بازی
- انتخاب تیم‌های محبوب در چند رقابت
- پیش‌بینی مسابقات و چالش‌های هفتگی
- لیدربورد جهانی، رقابتی و دوستانه
- پروفایل فوتبالی کاربر بر پایه Telegram ID

اصل طراحی محصول باید «ساده برای کاربر، چندلایه برای سیستم» باشد؛ یعنی کاربر فقط رقابت، تیم و پیش‌بینی را می‌بیند، اما backend باید بتواند چند لیگ، فصل، provider و قوانین امتیازدهی را مدیریت کند.

## ۲. بخش‌های اصلی اپلیکیشن

تب‌های پیشنهادی آینده:

- خانه
- زنده
- رقابت‌ها
- پیش‌بینی
- لیدربورد
- تیم‌های من
- پروفایل

توضیح نقش هر بخش:

- خانه: خلاصه شخصی‌سازی‌شده از بازی‌های مهم، تیم‌های محبوب، پیش‌بینی‌های باز و نتایج اخیر.
- زنده: بازی‌های در جریان در همه رقابت‌های فعال، همراه با دقیقه، نتیجه، رویدادها و وضعیت.
- رقابت‌ها: لیست لیگ‌ها و تورنمنت‌ها، فصل‌ها، جدول‌ها، برنامه بازی‌ها و فیلترها.
- پیش‌بینی: مسابقات قابل پیش‌بینی، وضعیت قفل‌شدن، امتیازهای احتمالی و تاریخچه پیش‌بینی‌ها.
- لیدربورد: رتبه‌بندی کلی، رتبه‌بندی هر رقابت، هفتگی و در آینده دوستان.
- تیم‌های من: تیم‌های محبوب کاربر در رقابت‌های مختلف و تنظیمات اعلان.
- پروفایل: اطلاعات Telegram، زبان، آمار کاربر، امتیازها و تنظیمات حساب.

## ۳. معماری Backend

### MatchPulse App Backend

لایه اصلی اپلیکیشن که API عمومی مینی‌اپ را ارائه می‌دهد. این لایه نباید به جزئیات provider وابسته باشد و باید فقط داده نرمال‌شده مصرف کند.

مسئولیت‌ها:

- احراز هویت سبک بر پایه Telegram ID
- مدیریت کاربران، علاقه‌مندی‌ها، یادآورها و پیش‌بینی‌ها
- ارائه API پایدار به frontend
- ترکیب داده مسابقات با داده کاربر
- حفظ رفتار سازگار بین local و server

### Data Provider / Wrapper Layer

لایه wrapper مسئول دریافت داده از منابع بیرونی است. MatchPulse نباید مستقیماً به providerهایی مثل Varzesh3، football-data یا هر منبع دیگر وابسته شود.

مسئولیت‌ها:

- دریافت داده خام از provider
- normalize کردن تیم‌ها، بازی‌ها، وضعیت‌ها، رویدادها، جدول‌ها و ویدیوها
- نگه‌داشتن mapping داخلی provider
- ارائه API تمیز و قابل مصرف برای MatchPulse
- مخفی‌کردن تفاوت‌های providerها از core app

قانون مهم: معماری wrapper-first باید حفظ شود.

### Competition Engine

لایه‌ای برای مدل‌کردن رقابت‌ها، فصل‌ها و قوانین اختصاصی هر رقابت.

مسئولیت‌ها:

- تعریف competition_key و season_key
- مدیریت مرحله‌ها، گروه‌ها، جدول‌ها و فرمت حذفی
- تشخیص وضعیت فصل
- نگهداری قوانین خاص مثل صعود، پلی‌آف، وقت اضافه و پنالتی

### Prediction Engine

لایه مسئول ثبت، قفل‌کردن، محاسبه و امتیازدهی پیش‌بینی‌ها.

مسئولیت‌ها:

- تعریف قوانین پیش‌بینی برای هر رقابت
- قفل‌کردن پیش‌بینی قبل از kickoff
- محاسبه امتیاز بعد از پایان بازی
- جلوگیری از ثبت تکراری یا تغییر غیرمجاز
- پشتیبانی از چالش‌های هفتگی و رقابتی

### Notification Engine

لایه مسئول اعلان‌ها برای یادآورها، شروع بازی، گل‌ها، پایان بازی و نتایج پیش‌بینی.

مسئولیت‌ها:

- ارسال یادآورهای قبل از بازی
- ارسال اعلان گل فقط برای رویدادهای scoring معتبر
- ارسال اعلان پایان بازی فقط بعد از تایید نهایی
- جلوگیری از اعلان تکراری
- احترام به تنظیمات کاربر و تیم‌های محبوب

### User / Profile Layer

لایه مدیریت کاربران و ارتباط آن‌ها با رقابت‌ها، تیم‌ها، پیش‌بینی‌ها و اعلان‌ها.

مسئولیت‌ها:

- ذخیره Telegram ID، نام، username و زبان
- مدیریت تیم‌های محبوب در سطح عمومی یا رقابت خاص
- نگهداری پروفایل کاربر برای هر competition
- آمار پیش‌بینی، امتیاز و رتبه کاربر

## ۴. پیش‌نویس مدل داده

جدول‌ها یا collectionهای پیشنهادی:

- competitions: اطلاعات رقابت‌ها مثل key، نام، کشور، نوع رقابت و وضعیت فعال بودن.
- seasons: فصل‌های هر رقابت، بازه زمانی، وضعیت و provider mapping.
- teams: تیم‌ها با نام فارسی، انگلیسی، لوگو، کشور و شناسه‌های provider.
- matches: بازی‌ها با شناسه داخلی پایدار، competition_key، season_key، تیم‌ها، kickoff، وضعیت و نتیجه.
- standings: جدول رقابت یا گروه با بازی، برد، مساوی، باخت، گل زده، گل خورده، تفاضل و امتیاز.
- match_events: رویدادهای نرمال‌شده بازی شامل گل، کارت، تعویض، VAR، پنالتی و ویدیو.
- users: اطلاعات پایه کاربر Telegram.
- favorite_teams: تیم‌های محبوب کاربر، ترجیحاً با امکان scope بر اساس competition.
- predictions: پیش‌بینی‌های ثبت‌شده کاربر برای هر بازی.
- prediction_rules: قوانین امتیازدهی هر رقابت یا نوع پیش‌بینی.
- prediction_scores: امتیازهای محاسبه‌شده برای هر پیش‌بینی.
- leaderboards: رتبه‌بندی‌های آماده یا cache شده برای نمایش سریع.
- user_competition_profiles: وضعیت کاربر در هر رقابت، امتیاز، رتبه، تیم محبوب و تنظیمات.

## ۵. پیش‌نویس طراحی API

Endpointهای پیشنهادی:

- `GET /competitions`
- `GET /competitions/{competition_key}/seasons`
- `GET /competitions/{competition_key}/matches`
- `GET /competitions/{competition_key}/standings`
- `GET /matches/{match_id}/live`
- `GET /matches/{match_id}/events`
- `GET /predictions/rules`
- `GET /leaderboards`
- `GET /users/{telegram_id}/profile`

نکته طراحی: مسیرهای فعلی مثل `/matches` و `/worldcup/summary` باید در مراحل اولیه حفظ شوند. endpointهای جدید باید در کنار مسیرهای فعلی اضافه شوند، نه به جای آن‌ها.

## ۶. طراحی Prediction v2

انواع پیش‌بینی پیشنهادی:

- پیش‌بینی نتیجه 1X2: برد میزبان، مساوی، برد مهمان
- پیش‌بینی نتیجه دقیق
- over/under تعداد گل‌ها
- گل‌زدن هر دو تیم
- تیم صعودکننده در بازی‌های حذفی
- روش صعود: ۹۰ دقیقه، وقت اضافه یا پنالتی
- چالش‌های هفتگی پیش‌بینی

مدل امتیازدهی پیشنهادی:

- تشخیص درست برنده: ۳ امتیاز
- تشخیص درست مساوی: ۴ امتیاز
- نتیجه دقیق: ۷ امتیاز
- bonus تفاضل گل درست: ۲ امتیاز
- bonus مجموع گل درست: ۲ امتیاز
- bonus روش صعود در حذفی: ۲ امتیاز
- ضریب پیش‌بینی زودهنگام می‌تواند در فازهای بعدی بررسی شود

قوانین مهم:

- هر پیش‌بینی باید قبل از kickoff قفل شود.
- تغییر پیش‌بینی بعد از قفل‌شدن مجاز نیست.
- scoring باید idempotent باشد تا با اجرای چندباره scheduler امتیاز تکراری ثبت نشود.
- قوانین هر رقابت باید قابل تنظیم باشد، نه hardcode شده در UI.

## ۷. طراحی لیدربورد

انواع لیدربورد پیشنهادی:

- لیدربورد جهانی برای کل کاربران
- لیدربورد هر رقابت
- لیدربورد هفتگی
- لیدربورد دوستان در آینده
- لیدربورد هواداران یک تیم محبوب در آینده

نکات اجرایی:

- محاسبه لیدربورد بهتر است cache شود.
- رتبه کاربر باید سریع و کم‌هزینه قابل دریافت باشد.
- leaderboard_key باید شامل competition_key، season_key و بازه زمانی باشد.
- frontend باید بتواند رتبه کاربر را کنار رتبه‌های برتر نمایش دهد.

## ۸. استراتژی مهاجرت

### Phase 0: تثبیت وضعیت فعلی

آرشیو جام جهانی ۲۰۲۶ پایدار بماند و فقط bugfix انجام شود. مسیرهای فعلی مثل `/matches` و `/worldcup/summary` تغییر رفتاری نداشته باشند.

### Phase 1: پایه backend چندرقابتی

مدل competition، season و team mapping اضافه شود، بدون تغییر رفتار frontend فعلی. داده‌های جام جهانی همچنان از مسیر فعلی قابل مصرف باشند.

### Phase 2: APIهای competition-aware

endpointهای جدید رقابت‌محور اضافه شوند و frontend به‌تدریج competition selector بگیرد. مسیر قدیمی `/matches` برای سازگاری باقی بماند.

### Phase 3: ساخت Prediction v2

prediction engine جدید با ruleهای قابل تنظیم ساخته شود. prediction فعلی تا زمان مهاجرت کامل نباید شکسته شود.

### Phase 4: لیدربورد

امتیازها به leaderboardهای جهانی، رقابتی و هفتگی وصل شوند. cache و jobهای محاسبه رتبه اضافه شوند.

### Phase 5: افزودن رقابت‌ها و providerهای بیشتر

رقابت‌های بیشتر و wrapperهای جدید اضافه شوند. core app همچنان باید فقط داده نرمال‌شده مصرف کند و به provider internals وابسته نشود.

## ۹. قوانین مهندسی مهم

- آرشیو فعلی جام جهانی نباید شکسته شود.
- جدول‌های قدیمی در مراحل اولیه migration حذف نشوند.
- منطق provider باید بیرون از core app باقی بماند.
- برای هر رقابت از `competition_key` پایدار استفاده شود.
- برای هر فصل از `season_key` پایدار استفاده شود.
- رفتار local و server باید یکسان باشد.
- main app نباید مستقیماً به جزئیات provider وابسته شود.
- معماری wrapper-first اولویت دارد.
- از بازنویسی big-bang پرهیز شود.
- تغییرها باید مرحله‌ای، قابل تست و قابل rollback باشند.
- prediction و notification باید idempotent طراحی شوند.

## ۱۰. جمع‌بندی

MatchPulse v2 باید با حفظ ارزش فعلی جام جهانی، به یک پلتفرم چندرقابتی تبدیل شود. مسیر امن این است که ابتدا backend foundation و قراردادهای داده ساخته شوند، سپس frontend به‌تدریج competition-aware شود. تا وقتی foundation کامل نشده، تجربه فعلی کاربر باید پایدار و بدون تغییر ناخواسته باقی بماند.
